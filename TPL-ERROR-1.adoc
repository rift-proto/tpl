TPL-ERROR-1
============
Error Codes
-----------
`mandatory` `server` `client`

This TPL defines the structured error codes returned in Error frames
(`frame_type = E`) in the Rift Realtime Protocol (Rift/1).

Error codes are transmitted as stable `RIFT_*` string identifiers so that
older clients can gracefully handle unknown codes without breaking.

== Protocol Errors

[cols="1,3,1"]
|===
| Code | Description | Retryable

| RIFT_PROTOCOL_VERSION_UNSUPPORTED
| Requested protocol version is not supported by the server.
| No

| RIFT_PROTOCOL_FRAME_INVALID
| Received frame is structurally invalid (malformed header, missing required fields).
| No

| RIFT_PROTOCOL_CODEC_UNSUPPORTED
| Requested content codec is not supported.
| No

| RIFT_PROTOCOL_PAYLOAD_TOO_LARGE
| Frame payload exceeds the server's configured maximum size.
| No

| RIFT_PROTOCOL_REQUIRED_FIELD_MISSING
| A required field is missing from the frame.
| No

| RIFT_PROTOCOL_SCHEMA_MISMATCH
| Frame contents do not match the expected schema.
| No

| RIFT_PROTOCOL_ORDER_VIOLATION
| Frames were received out of the required order (e.g. Publish before Subscribe).
| No
|===

== Retryability

Implementations SHOULD retry with exponential backoff for retryable errors.
Non-retryable errors indicate permanent failures that require corrective action
(obtaining fresh credentials, fixing the frame, upgrading the SDK, etc.).

== Conformance

Implementations MUST:

- Transmit error codes as their stable `RIFT_*` string identifier.
- Handle unknown error codes gracefully (log and display, do not panic).
- Include a human-readable description in the error frame payload.
