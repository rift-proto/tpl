TPLs — Rift Implementation Possibilities
=========================================

TPL stands for **TPL — Technical Protocol Listing**.
TPLs exist to document what must, should, or may be implemented by Rift-compatible
_server_ and _client_ software.

Each TPL is a focused specification document covering a single aspect of the
Rift Realtime Protocol (Rift/1). They are designed to be read independently
while cross-referencing each other where necessary.

== Protocol Overview

The Rift Realtime Protocol (Rift/1) is a modern, real-time bidirectional messaging
protocol designed to replace raw WebSocket + event-string models with stronger semantics.

=== Key Features

* Typed frame envelope with priority and flags
* CBOR + JSON codec negotiation
* Eight message classes (Event, Command, Reply, State, Datagram, Stream, Snapshot, System)
* Topic-based pub/sub with retention and ordering policies
* Session management with auth and disconnect-resume
* Acknowledgement system with 9 status levels and 6 policies
* Backpressure control (6 strategies) and token-bucket rate limiting
* Heartbeat with jitter for thundering-herd prevention

=== Connection Lifecycle

```text
WebSocket Open
  → Hello (client → server)
  → Welcome (server → client)
  → [Resume exchange, optional]
  → Ready (server → client)
  → [Data frames flow bidirectionally]
  → Close (either side)
```
